HashMap

  • HashMap 的数据结构

底层是数组 + 链表，在 jdk 8 中，当链表长度大于等于 8 时会转换为红黑树。

  • HashMap 的工作原理

调用 put 方法时，会计算 key 的 hash 值（hashCode 与 hashCode 无符号右移 16 位进行异或），然后获取数组下标也就是桶号（(n - 1) & hash），如果该下标节点为空，那么新建一个节点，如果节点不为空，那么遍历单链表，如果遇到与插入 key 相等的节点，那么更新值，否则创建一个节点插入到链表尾端，如果元素数量到 8 了，那么转换为红黑树，最后再判断下元素个数是否超出临界值（负载因子 * 容量），如果超出需要进行扩容。

  • 你知道 hash 的实现吗？为什么要这样实现？

将 hashCode 于其高 16 位进行按位异或。因为一般情况下 map 大小都小于 65536，因此计算下标时也只考虑了 hash 的低 16 位，高位没有参与下标计算，会增加 hash 碰撞概率。

  • HashMap 的 table 的容量如何确定？

容量固定为 2 的幂，默认容量为 16，需要等到插入元素时才会扩容成 16。

  • HashMap 的 loadFactor 是什么？ 该容量如何变化？这种变化会带来什么问题？

用于确定是否需要扩容，当元素个数大于 loadFactor * capacity 时就需要进行扩容，扩容为原先容量的两倍。

  • HashMap 是如何进行扩容的？

扩容时数组容量变为两倍，但是只需要将新增的那位是 1 的元素索引改为原来索引 + 原来容量就行。原来是 16 10000，现在改为 32 100000，计算下标方法为 (n - 1) & hash，也就是 1111 & hash 变为 11111 & hash，那么如果 hash 低 5 位为 0 ，那么结果就不发生变化，否则需要加 16。

  • HashMap 容量为什么要是 2 的幂？

首先是高效的计算数组下标只需使用（n - 1）& hash 就行了进行取余操作，其次扩容时也只需要移动一半元素就行了。

  • HashMap 线程是否安全？

线程不安全，在进行扩容时多线程可能导致死锁，HashTable 是线程安全的，一般而言可以使用 ConcurrentHashMap，因为其使用了分段锁，没段一把锁，这样就支持多线程访问。
